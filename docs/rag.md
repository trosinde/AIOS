# Retrieval-Augmented Generation (RAG)

> **Audience:** Users + Developers

## Overview

AIOS includes a built-in RAG system that adds semantic search over custom data collections. Instead of relying solely on the LLM's training data, you can index your own documents and retrieve them by meaning at query time.

Key capabilities:
- **Index** structured or unstructured documents into named collections
- **Search** collections using natural language queries
- **Compare** documents across collections for similarity
- **Integrate** RAG steps into automated workflows via the Router

## How It Works

The RAG pipeline has three components:

1. **Embedding Provider** — Converts text into vector representations. Two options:
   - `local` — Runs Transformers.js in-process (default: `Xenova/all-MiniLM-L6-v2`). No external service needed.
   - `ollama` — Uses a running Ollama instance with an embedding model (e.g., `all-minilm`).

2. **Vector Store** — An in-memory store that holds document embeddings and performs cosine-similarity search. Documents are stored per collection with metadata.

3. **Preprocessing** — Before embedding, documents are chunked and cleaned according to collection config. Chunking strategies control how long documents are split into indexable units.

## Configuration

Add a `rag` section to `aios.yaml`:

```yaml
rag:
  defaultProvider: local                # "local" (Transformers.js) or "ollama"
  defaultModel: Xenova/all-MiniLM-L6-v2

  ollama:                               # Required only when using ollama provider
    model: all-minilm
    endpoint: http://localhost:11434

  collections:
    my-docs:
      preprocessing:
        maxChunkLength: 500             # Max characters per chunk
        chunkStrategy: truncate         # truncate | sliding_window
        chunkOverlap: 200              # Overlap size (for sliding_window only)
        cleaners:                       # Text cleaning pipeline
          - stripHtml
          - normalizeWhitespace
        fields: [title, description]    # Which JSON fields to embed (for structured data)
      search:
        minRelevance: 0.3               # Minimum cosine similarity (0-1)
        topK: 20                        # Max results returned
        queryExpansion:                  # Optional: expand abbreviations in queries
          DD: [Density Determination]
```

### Preprocessing Options

| Field | Type | Description |
|-------|------|-------------|
| `maxChunkLength` | number | Maximum characters per chunk |
| `chunkStrategy` | string | `truncate` (cut at limit), `sliding_window` (overlapping windows) |
| `chunkOverlap` | number | Characters of overlap between windows (sliding_window only) |
| `cleaners` | string[] | Cleaning steps: `stripHtml`, `normalizeWhitespace` |
| `fields` | string[] | For JSON input: which object fields to concatenate for embedding |
| `fieldSeparator` | string | Separator between concatenated fields (default: space) |

### Search Options

| Field | Type | Description |
|-------|------|-------------|
| `minRelevance` | number | Minimum cosine similarity score (0.0 to 1.0) |
| `topK` | number | Maximum number of results to return |
| `queryExpansion` | Record | Map abbreviations to expanded terms for query enrichment |

## RAG Patterns

AIOS provides two internal patterns for RAG operations:

### `rag_search` — Semantic Search

Takes a natural language query and returns the most relevant documents from a collection.

```bash
echo "authentication requirements" | aios run rag_search
```

The Router sets the target collection via the `rag_collection` field in the execution plan step.

### `rag_index` — Document Indexing

Takes a JSON array of items and indexes them into a collection. Each item needs at minimum an `id` and either `content` or the fields specified in the collection config.

```bash
cat items.json | aios run rag_index
```

Input format:
```json
[
  { "id": "DOC-001", "title": "Auth Design", "description": "JWT-based auth flow..." },
  { "id": "DOC-002", "title": "API Gateway", "description": "Rate limiting and routing..." }
]
```

## RAG Operations

The RAG service supports three operations:

| Operation | Input | Output |
|-----------|-------|--------|
| `search` | Natural language query | Ranked list of matching documents with relevance scores |
| `index` | JSON array of items | Confirmation with count of indexed chunks |
| `compare` | Two collection references | Cross-collection similarity matrix |

## Usage in Workflows

The Router can automatically plan RAG steps when a task requires knowledge retrieval. A typical workflow:

1. **Search** the knowledge base for relevant context
2. **Feed** results into an analysis or generation pattern

Example execution plan (generated by Router):
```json
{
  "steps": [
    {
      "id": "find_docs",
      "pattern": "rag_search",
      "rag_collection": "my-docs",
      "input": "security requirements for API gateway"
    },
    {
      "id": "analyze",
      "pattern": "analyze_requirements",
      "input_from": ["find_docs"],
      "depends_on": ["find_docs"]
    }
  ]
}
```

## CLI Usage

Run RAG patterns directly:

```bash
# Search a collection
echo "search query here" | aios run rag_search

# Index documents from a JSON file
cat documents.json | aios run rag_index

# Use in a natural language task (Router plans RAG steps automatically)
aios "Find all requirements related to encryption in our docs"
```

The Router decides which collection to target based on the task description and the configured collection names.
